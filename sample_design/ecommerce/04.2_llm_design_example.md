# E-commerce LLM Design

**NOTE (for students):** This file is an example reference. Treat lines labeled `NOTE` as guidance; treat the remaining design content as an example implementation.

## 1. Goal

Design a simple LLM use case for e-commerce.

Suggested use case:

- customer support assistant for order and product questions

Main outcomes:

- answer user questions with grounded data
- reduce hallucination with retrieval and guardrails
- log and monitor quality, latency, and safety

**NOTE (for students):** Focus on system design and operations. You do not need a complex multi-agent design or an advanced model.

**Tooling constraint (required):** the system must use a tool call to a prediction API connected to the feature store.

**Input context profile (required before design)**

- Source scope: list trusted sources used for retrieval (docs/tables/apis) and why each source is allowed.
- Data volume: number of documents/chunks, average chunk size, and update frequency.
- Query profile: expected QPS/concurrency, peak periods, and common query types.
- Risk profile: sensitive fields, PII exposure risk, and unsafe prompt patterns.

Students must state this input profile first, then design agent behavior and serving components.

Example:

- Source scope: `faq_docs`, `return_policy_docs`, `gold_ecommerce.obt_order_performance_view` (read-only).
- Data volume: 25k chunks, average 650 chars/chunk, updates every 30 minutes.
- Query profile: average 3 QPS, peak 12 QPS during campaigns.
- Risk profile: order_id/email as sensitive fields; prompt-injection attempts expected in free-text inputs.

**Assumptions**

- Business objective: provide grounded support answers that improve resolution speed and user experience.
- Decision usage: answers are assistive and should not execute sensitive actions automatically.
- Service level expectation: response quality, latency, and safety controls must meet defined targets.
- Explainability expectation: source citation is required when relevant context exists.
- Governance expectation: full compliance controls are out of scope for this phase, but minimum safety controls are required.

**SLA targets (example)**

- API latency: p95 <= 2.0s for standard questions.
- Retrieval latency: p95 <= 600ms.
- Grounded-answer rate: >= 90% on evaluation set.
- Safety block precision: >= 95% for known policy-violating prompts.

Given coursework compute constraints, you may adjust SLA targets and report final achieved values after implementation.

## 2. High-Level LLM System Design

End-to-end flow:

1. User sends a question.
2. Gateway authenticates and routes request.
3. Unified indexing pipeline keeps retrieval index/vector store updated and also handles rebuild/recovery (reindex mode) when triggers are met.
4. System executes one tool call to the feature-store prediction API (which is an API using features from the feature store to make prediction).
5. Retrieval layer fetches trusted context (documents/read-only views + vector index).
6. Prompt builder composes system prompt + user question + retrieved context + prediction output.
7. LLM generates answer.
8. Safety layer validates output.
9. Response is returned and logged.

Example:

- User asks: "Where is order O12345 and can I return it?"
- Retrieval returns policy + order context.
- Single prediction API call returns churn-risk score used for response tone/prioritization.
- LLM returns grounded answer with cited source ids.

Key decisions students should explain:

- model choice (hosted API vs self-hosted, prioritize reliability/cost)
- system components and boundaries (gateway, retrieval, prompt, model, safety, observability)
- tool access policy (allowed sources, read-only access, timeout, retry)
- single-tool policy (feature-store prediction API only): request/response contract, timeout, retry, fallback on API failure
- retrieval strategy (dense, sparse or hybrid search)
- latency/cost trade-off
- safety and compliance controls
- fallback behavior when confidence is low (clarify/abstain/escalate)

**NOTE (for students):** The decisions above are examples, not exhaustive. Add other decisions relevant to your use case.

## 3. Low-Level LLM System Design

Move from abstract blocks to concrete interfaces and schemas.

Example key classes:

```python
class RequestService:
    def handle_request(self, user_id: str, question: str) -> dict: ...
    def build_response(self, result: dict) -> dict: ...

class RetrievalService:
    def search(self, query: str, top_k: int) -> list[dict]: ...
    def rerank(self, query: str, docs: list[dict]) -> list[dict]: ...

class PredictionToolService:
    def call_prediction_api(self, user_id: str, query_text: str, filters: dict) -> dict: ...
    def validate_prediction_response(self, payload: dict) -> None: ...

class DataAccessService:
    def read_knowledge_view(self, query: str) -> list[dict]: ...
    def validate_read_only_access(self, source: str) -> None: ...

class PromptService:
    def build_system_prompt(self) -> str: ...
    def build_user_prompt(self, question: str, context: list[dict]) -> str: ...

class LLMService:
    def generate(self, prompt: str) -> dict: ...
    def estimate_tokens(self, prompt: str) -> int: ...

class MemoryService:
    def get_session_context(self, user_id: str) -> dict: ...
    def save_turn(self, user_id: str, turn: dict) -> None: ...

class SafetyService:
    def validate_input(self, question: str) -> None: ...
    def validate_output(self, answer: str) -> None: ...

class EvaluationService:
    def score_grounding(self, answer: str, sources: list[dict]) -> float: ...
    def score_helpfulness(self, answer: str) -> float: ...

class ObservabilityService:
    def log_request(self, payload: dict) -> None: ...
    def publish_metrics(self, metrics: dict) -> None: ...
```

## 4. Data and Knowledge Sources Decision

Use trusted sources only:

- product catalog and policy docs
- FAQ and support playbooks
- selected Gold tables (read-only views)

Recommended table pattern in Gold:

- `llm_knowledge_chunks` (chunk_id, source_type, source_id, content, updated_ts) - stores retrievable text chunks used as LLM context.
- `llm_chunk_embeddings` (chunk_id, embedding_model, embedding_vector/ref, updated_ts) - stores vector representations for semantic retrieval.
- `llm_feedback` (request_id, rating, comment, created_ts) - stores user feedback for quality evaluation and iteration.

## 5. Indexing and Reindexing Pipeline Decision

Required jobs:

- chunking job: normalize and chunk trusted sources
- embedding job: generate embeddings for chunks
- index upsert job: write/update vector store index
- index validation job: check coverage, freshness, and retrieval sanity

Required reindex behavior (same pipeline, reindex mode):

- scheduled reindex: run periodically (for example daily or weekly) based on data update rate
- triggered reindex: run when quality/freshness/index-health thresholds are crossed
- candidate index validation: promote only if retrieval/evaluation checks pass
- rollback support: keep previous `index_version` and allow fast rollback

Example:

- Schedule: run chunking/embedding/upsert every 30 minutes.
- Validation: require coverage >= 99% of active chunks and freshness lag <= 45 minutes.

Example reindex triggers:

- new/updated chunk volume exceeds threshold
- retrieval precision@k drops below agreed baseline
- index sync lag exceeds freshness target
- embedding model version changes

Operational rules:

- Build candidate index from latest chunk + embedding snapshot.
- Validate candidate index with retrieval sanity checks and offline query set.
- Promote only if candidate passes quality/freshness checks.
- Keep rollback path to previous `index_version`.

Example:

- Trigger: precision@5 dropped from 0.80 to 0.68.
- Action: rebuild candidate `index_version=idx_20260407_b`.
- Validation: candidate reaches precision@5 = 0.79 on offline set.
- Decision: promote candidate; keep `idx_20260401_a` as rollback version.

## 6. Vector Store Design Decision

Required design points:

- index key: `chunk_id` and source metadata for filtering
- metadata filters: source_type, product/category, policy_type, language, updated_ts
- update mode: incremental upsert + periodic rebuild window
- retrieval policy: top_k, similarity metric, optional hybrid search with keyword filter

Example:

- Index key: `chunk_id`.
- Filters used at query time: `source_type=policy`, `language=en`, `updated_ts >= now-90d`.
- Retrieval setup: hybrid search with `top_k=8` and cosine similarity rerank.

## 7. Tool Call Contract Decision

Required contract:

- Use a tool call to a feature-store prediction API.
- Define input schema (for example: user_id, query_text, context_filters).
- Define output schema (for example: prediction label/score, confidence, model_version, timestamp).
- Define failure policy (timeout, retry, fallback response when tool is unavailable).

Example:

- Input: `{user_id, query_text, customer_segment, locale}`
- Output: `{prediction_label, prediction_score, confidence, model_version, score_ts}`
- Failure handling: timeout at 500ms, retry once, then continue with rule-based fallback message.

## 8. Prompt and Retrieval Pattern Decision

Simple prompt structure:

- system prompt: role, policy, and tone
- context block: retrieved chunks with source ids
- user prompt: question

Simple request loop:

1. retrieval service fetches trusted context
2. prediction API tool call returns prediction output
3. prompt service builds final prompt
4. LLM generates answer
5. evaluator/safety checks before returning output

Response rules:

- cite source ids when possible
- if confidence is low, say "I do not have enough context"
- do not invent order status, prices, or policy details

Example:

- Context sources: `policy_17`, `order_view_row_8842`
- Output: "Based on `policy_17`, returns are allowed within 30 days. For order status, `order_view_row_8842` shows shipment in transit."

## 9. Serving Pattern Decision

Recommended start:

- synchronous API for chat requests
- cache frequent queries
- timeout and retry policy
- read-only retrieval access to vector store and knowledge views

Scale options:

- async queue for long tasks
- model fallback chain (small model -> larger model)
- background indexing pipeline schedule (for example every 15-60 minutes)

Example:

- API service: FastAPI with `/chat` endpoint, p95 timeout set to 2.5s.
- Cache: 5-minute cache for repeated FAQ-style questions.
- Fallback: if primary model times out, call smaller backup model once.
- Indexing schedule: run chunking/embedding/index-upsert every 30 minutes.

## 10. Security and Safety Decision

Minimum controls:

- API auth and RBAC
- secret management for model keys
- PII masking in logs
- prompt injection checks
- output filtering for harmful or policy-violating content

Example:

- Only authenticated support users can call `/chat` (JWT + RBAC).
- API keys are read from secret manager, never hardcoded.
- Email/phone/order_id are masked in request/response logs.
- Prompt-injection rule blocks instructions such as "ignore previous policy".
- Safety filter blocks disallowed content before returning response.

## 11. Monitoring Decision

Track:

- quality: answer helpfulness and user rating
- reliability: error rate and timeout rate
- performance: p95 latency and throughput
- cost: token usage by route and model
- safety: blocked prompts and blocked outputs

Also collect:

- logs, metrics, traces

**Observability requirements (required):**

- Logs: required for debugging API/pipeline behavior and compute issues.
- Metrics: track latency, timeout rate, token usage, tool-call success rate, safety block rate, and compute-resource metrics (CPU, memory, disk I/O, network I/O).
- Traces: capture end-to-end spans for gateway -> retrieval -> prediction tool call -> prompt build -> LLM generate -> safety check.

Example:

- Quality alert: weekly average rating < 3.8/5.
- Reliability alert: timeout rate > 3% for 10 minutes.
- Performance alert: p95 latency > 2.5s.
- Cost alert: daily token usage exceeds planned budget by 20%.
- Safety alert: blocked-output rate spikes above baseline.
- Log requirement: use structured JSON logs and dedicated indices for API/pipeline debugging (see Section 12).
- Trace example: one trace with spans `gateway`, `retrieval`, `prediction_api`, `llm_generate`, `safety_filter`.

## 12. Application Log Storage Decision (Debugging and Investigation)

Scope rule:

- This section covers application logs only (LLM request/tool/pipeline/safety logs).
- Exclude infrastructure/system logs (OS/container/node/service mesh platform logs).

Log index design (Elasticsearch/OpenSearch example):

Use separate indices by log type for easier retention and querying.

1. `llm-query-logs-YYYY.MM.DD`

- Purpose: request/response observability and latency/cost debugging.
- Log format: JSON, one event per line.
- Required fields: `@timestamp`, `request_id`, `user_id_hash`, `route`, `model_version`, `index_version`, `latency_ms`, `tool_status`, `tokens_prompt`, `tokens_completion`, `tokens_total`, `safety_action`, `http_status`, `error_code`.

2. `llm-safety-logs-YYYY.MM.DD`

- Purpose: safety decisions and policy enforcement audit.
- Log format: JSON, one event per line.
- Required fields: `@timestamp`, `request_id`, `policy_version`, `injection_detected`, `blocked`, `blocked_reason`, `risk_score`, `action_taken`.

3. `llm-pipeline-logs-YYYY.MM.DD`

- Purpose: indexing/reindexing pipeline run diagnostics.
- Log format: JSON, one event per line.
- Required fields: `@timestamp`, `pipeline_name`, `run_id`, `stage`, `status`, `input_count`, `output_count`, `duration_ms`, `index_version`, `error_summary`.

Example application log documents:

```json
{"@timestamp":"2026-04-07T10:12:00Z","request_id":"req_123","user_id_hash":"u_9ab","route":"/chat","model_version":"v2.1","index_version":"idx_20260407","latency_ms":842,"tool_status":"success","tokens_prompt":730,"tokens_completion":120,"tokens_total":850,"safety_action":"allow","http_status":200}
{"@timestamp":"2026-04-07T10:12:01Z","request_id":"req_123","policy_version":"p3","injection_detected":false,"blocked":false,"risk_score":0.08,"action_taken":"allow"}

```

Retention recommendation:

- query logs: 14-30 days hot, then warm/archive
- safety logs: 30-90 days for audit
- pipeline logs: 14-30 days

## 13. Evaluation Decision

Evaluate with:

- offline test set (known Q&A)
- retrieval quality (precision@k)
- response quality review rubric
- safety checks (prompt injection and policy tests)

Example:

- Offline set: 200 curated support questions across order status, returns, and policy queries.
- Retrieval target: precision@5 >= 0.75 on the offline set.
- Response rubric: grounding, correctness, clarity, and policy compliance (1-5 scale).
- Safety target: 100% block rate for known injection test prompts.

## 14. CI/CD for LLM Service and Pipelines Decision

CI/CD layout (example):

Track A: **CI/CD for LLM API Service**

- CI: lint/tests, prompt template checks, retrieval contract checks, and safety regression tests.
- CD: deploy service and prompt/config bundles across `dev -> staging -> production`.
- Release controls: canary rollout, health checks, and rollback on quality/safety regression.

Track B: **CI/CD for All Pipelines (ML-style coverage)**

- CI: pipeline unit tests, DAG/config validation, schema checks, and data-quality tests for all pipelines used by this LLM system.
- CD: deploy all required pipelines across `dev -> staging -> production`.
- Include LLM-specific pipelines explicitly: chunking, embedding, index-upsert, and index-validation.
- Release controls: schedule validation, freshness SLO checks, rollback/re-run plan for failed pipeline releases.

Example:

- Pipeline CI jobs: `ci_chunking`, `ci_embedding`, `ci_index_upsert`, `ci_index_validation`.
- Pipeline CD jobs: `cd_chunking`, `cd_embedding`, `cd_index_upsert`, `cd_index_validation`.
- Promotion rule: all pipeline CI jobs pass in staging before production rollout.

**NOTE (for students):** CI/CD coverage is required for the API service and all pipelines, including LLM-specific indexing pipelines.
