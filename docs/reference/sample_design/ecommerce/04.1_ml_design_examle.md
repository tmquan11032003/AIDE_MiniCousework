# E-commerce ML Design

**NOTE (for students):** This file is an example reference. Treat lines labeled `NOTE` as guidance; treat the remaining design content as an example implementation.

## 1. Goal

Build a simple ML use case on top of Gold data and feature tables.

**Suggested task:** predict whether a customer will purchase in the next session.

**Why this task:** it uses offline + streaming features.

---

## 2. Prediction Setup and Modeling Plan

**Prediction setup**

- Entity: customer_id
- Label: `will_purchase_next_session` (1 = yes, 0 = no)
- Prediction time: use the feature snapshot at `event_timestamp`

**Training data input**

- Table: `ml_customer_purchase_training` (created in 03_data_generator_with_drift.md, Gold zone)
- Grain: one row per customer_id per timestamp
- Columns: customer_id, event_timestamp, created_ts, label, and feature columns from Gold feature tables
- Use: model training input

**Modeling decisions**

- Split strategy: choose based on problem and data leakage risk (time-based, random/stratified, etc.)
- Baseline model: choose based on problem type and constraints (for example, Logistic Regression, Decision Tree)
- Metrics: choose based on business objective and class balance (for example, Precision, Recall, F1, PR-AUC)

**Assumptions**

- Business objective: improve conversion by prioritizing likely-to-purchase customers.
- Decision usage: scores are used for ranking/personalization support, not as fully automated hard decisions.
- Service level expectation: prediction results must be available within agreed latency/freshness targets.
- Explainability expectation: out of scope for the current phase.
- Risk and governance expectation: out of scope for the current phase.

**SLA targets (example)**

- Online inference latency: p95 < 200 ms.
- Stream feature freshness: < 5 minutes.
- Batch score freshness: < 24 hours.

Given coursework compute constraints, you may adjust SLA targets and report the final achieved values after completing the implementation.

## 3. High-Level ML Design

End-to-end flow:

1. Build and validate `ml_customer_purchase_training` from Gold feature/label inputs.
2. Run the training pipeline (dataset build -> train -> evaluate -> register model).
3. Run the retrain-trigger pipeline (drift/quality trigger -> retrain -> evaluate -> promote or rollback).
4. Choose and implement one serving pattern (online, stream, or precomputed table) and return prediction scores.
5. Run CI/CD pipelines for data/ML jobs and inference services.
6. Monitor logs, metrics, and traces for compute resources, data pipelines, and ML services.

**NOTE (for students):** See Section 7 for pipeline details, Section 8 for CI/CD pipeline design, Section 9 for monitoring, and Section 10 for retraining strategy.

Other flows (example):

- **IaC setup flow:** provision/update environments and shared platform resources using versioned templates.
- **Versioning:** version data snapshots, features, model artifacts, and config files.

**NOTE (for students):** The decision areas below are examples, not a fixed or exhaustive list. Add other relevant decisions for your use case.

### 3.1 Security Decision

Example decision focus:

- How you protect training data, model artifacts, and inference endpoints.

Example considerations:

- IAM/RBAC for least-privilege access.
- Secrets management for credentials and tokens.
- Encryption at rest and in transit.

Example:

- Only the training service account can read `ml_customer_purchase_training`.
- Model registry is write-restricted to CI/CD service identity.
- Inference API requires basic auth.

### 3.2 Resilience Decision

Example decision focus:

- How the system handles failures, retries, and safe recovery.

Example considerations:

- Retry + backoff policy for pipeline jobs.
- Idempotent job behavior for re-runs/backfills.
- Rollback strategy for model and service releases.

Example:

- `train_job` retries up to 3 times with exponential backoff.
- `scoring_job` is partition-idempotent to avoid duplicate writes.
- Failed canary deployment auto-rolls back to previous `model_version`.

### 3.3 Training/Inference Pattern Decision

Example decision focus:

- Why your chosen training and serving mode match the use case.

Example considerations:

- Training mode: scheduled, triggered, or hybrid.
- Inference mode: online, stream, or precomputed batch.
- Trade-offs: latency, freshness, cost, and ops complexity.

Example:

- Weekly scheduled training + drift-triggered retraining.
- Stream inference for near-real-time promotions.
- Batch precompute fallback for dashboard/reporting use cases.

### 3.4 Storage Decision

Example decision focus:

- Where data/features/models/logs are stored and why.

Example considerations:

- Bronze/Silver/Gold tables for lifecycle separation.
- Feature storage strategy (batch + low-latency serving needs).
- Model registry and log retention policy.

Example:

- Bronze/Silver in lakehouse tables (Delta Lake) and Gold in DWH (PostgreSQL).
- Curated feature tables in Gold for training; cached recent features for online scoring.
- Model artifacts in registry + object storage with 90-day retention for logs.

### 3.5 Routing/Gateway Decision

Example decision focus:

- How requests are routed to inference services safely and efficiently.

Example considerations:

- API gateway or ingress pattern.
- Rate limiting, authentication, and request validation.
- Versioned routing (blue/green or canary).

Example:

- API gateway routes `/score` to inference service v2.
- 10% canary traffic to new model endpoint before full rollout.
- Rate limits protect the service during traffic spikes.

## 4. Low-Level ML Design

Low-Level Design (LLD) for ML focuses on concrete class structures, interfaces, and data contracts instead of abstract boxes like "Model" or "Database".

**NOTE (for students):** For this ecommerce task, propose 5 key classes. Each class can have multiple functions.
Learn more about Design Pattern here: https://github.com/msaroufim/ml-design-patterns

```python
class TrainingDataService:
    def read_training_table(self) -> DataFrame: ...
    def validate_schema(self, df: DataFrame) -> None: ...
    def dedup_by_created_ts(self, df: DataFrame) -> DataFrame: ...

class SplitService:
    def get_split_boundaries(self, df: DataFrame) -> dict: ...
    def split_by_time(self, df: DataFrame) -> tuple[DataFrame, DataFrame, DataFrame]: ...

class ModelService:
    def train(self, train_df: DataFrame) -> ModelArtifact: ...
    def evaluate(self, model: ModelArtifact, test_df: DataFrame) -> dict: ...
    def save_model(self, model: ModelArtifact, model_version: str) -> None: ...

class ScoringService:
    def score_online(self, model: ModelArtifact, request: dict) -> dict: ...
    def score_stream(self, model: ModelArtifact, event: dict) -> dict: ...
    def score_batch(self, model: ModelArtifact, df: DataFrame) -> DataFrame: ...
    def write_scores(self, score_df: DataFrame, model_version: str) -> None: ...

class MonitoringService:
    def publish_model_metrics(self, metrics: dict) -> None: ...
    def publish_drift_metrics(self, drift_stats: dict) -> None: ...
    def trigger_alerts(self, metrics: dict) -> None: ...
```

## 6. Infrastructure Decision (ML + Data Job Scaling)

Infrastructure options (example):

- ML workloads: training, inference, retraining
- Data jobs: ingestion, transformation, feature computation, and batch scoring

1. **Kubernetes (K8s)**

- Good for scaling, self-healing, and multi-service workloads.
- Suitable for online/stream serving and scalable data/feature pipelines.

2. **Virtual Machine (VM)**

- Simpler for small teams and lower scale.
- Suitable for batch ETL, scheduled retraining, and lower-throughput serving.

3. **Hybrid**

- Example: data pipelines and training on batch cluster/VM, online serving on K8s.

Node resource planning (minimum should be justified by workload):

- CPU (vCPU)
- Memory (RAM)
- Storage (raw/intermediate data, model artifacts, feature cache, logs)

Example sizing approach:

- Start with data volume and job frequency (batch + streaming).
- Estimate compute for ETL/feature jobs and model inference separately.
- Add headroom for traffic spikes, backfills (historical re-runs), and retraining overlap.
- Define autoscaling policy (if K8s) or vertical scaling plan (if VM).

**NOTE (for students):** Sizing and serving choices should satisfy the SLA targets defined in Section 2.

- serving pattern (online/stream/precomputed, etc.)
- expected traffic profile and data job workload
- cost and operational complexity

## 7. Pipeline Design (Data + ML)

Pipeline layout (example):

**NOTE (for students): Scope for 04**

- All data pipelines (Bronze/Silver/Gold and feature pipelines) are designed and implemented in 02 and should be reused.
- In 04, implementation focus is ML pipelines (training, batch scoring, retraining) and inference services.

**NOTE (for students):** Configure pipeline schedules and alert thresholds to meet the SLA targets from Section 2.

1. **ML Pipeline Group**

Pipeline A: **Training Pipeline**

1. `training_dataset_job`
   - joins `ml_customer_label` with Gold feature tables by entity + `event_timestamp`
   - outputs `ml_customer_purchase_training`
2. `train_job`
   - trains baseline model and stores model artifact
3. `evaluation_job`
   - computes Precision/Recall/F1 and compares against acceptance threshold
4. `model_registry_job`
   - registers approved model with `model_version`

Pipeline B: **Batch Serving Pipeline**

1. `scoring_job` (batch mode)
   - reads batch input or scoring table
   - produces precomputed scores
2. `write_scores_job`
   - writes `customer_id`, `score`, `model_version`, `score_ts` to serving table

Pipeline C: **Retrain Trigger Pipeline**

1. `retrain_trigger_job`
   - checks drift and model quality thresholds (for example, F1 drop, feature drift spikes)
   - opens retrain event when threshold is crossed
2. `retrain_job`
   - rebuilds latest training window and retrains candidate model
3. `retrain_evaluation_job`
   - compares candidate vs current production model
4. `promotion_or_rollback_job`
   - promotes candidate if accepted, otherwise keeps/rolls back to previous `model_version`

CI/CD layout (example):

**NOTE (for students): Required implementation scope**

- You must build CI/CD for each implemented inference service.
- Minimum expectation: each pipeline/service has both CI checks and a CD/deployment flow.

**NOTE (for students):** CI/CD examples below show one reference structure. You may rename jobs/tools, but coverage per pipeline/service is required.

## 10. Retraining Strategy

Use a simple retraining policy:

- Scheduled retrain: once per week (or once per month for stable data).
- Triggered retrain: retrain when drift or model quality crosses threshold.

Example triggers:

- feature drift alert stays high for multiple days
- F1 drops below agreed baseline
- data volume/seasonality changes significantly

Operational rules:

- Train candidate model on latest training window.
- Compare candidate vs current production model.
- Promote only if candidate is better and passes quality checks.
- Keep rollback path to previous `model_version`.
